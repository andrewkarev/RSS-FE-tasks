Introduction
What is DRY?
"Don't Repeat Yourself"
or
"Duplication is evil"
DRY was first mentioned in the book
«The Pragmatic Programmer»
by Andrew Hunt and David Thomas, 1999
"Each piece of knowledge must have a single, monosemantic, reliable representation in the system"

Ways to combine code duplications

  • Methods
  • Modules
  • Components

DRY is based on:

  • Decomposition of tasks
  • Modular structure

Advantages of the DRY usage

  • The amount of code written reduces
  • Code becomes easier to refactor, change and make additions to it
  • Code becomes easier to maintain in the future
  • Code structure is simplified

Is it always appropriate to use DRY?

similar code VS code with identical logic

#### Functions 
            ``` js
            const monthNumber = Math.floor((Math.random() * 12) + 1);

            if (monthNumber < 3 || monthNumber === 12) {
              console.log("It is a winter month. Time to go skiing.");
            } else if (monthNumber >= 3 && monthNumber < 6) {
              console.log("It is a spring month. Time to skateboard.");
            } else if (monthNumber >= 6 && monthNumber < 9) {

            }
            ```
          
            ``` js
            const monthNumber = Math.floor((Math.random() * 12) + 1);
            const defineWhatItIsTimeFor = (timeOftheYear, action) => {
              console.log(`It is a ${timeOftheYear} month. Time to ${action}.`);
            }

            if (monthNumber < 3 || monthNumber === 12) {
              defineWhatIsItTimeFor("winter", "go skiing");
            } else if (monthNumber >= 3 && monthNumber < 6) {
              defineWhatIsItTimeFor("spring", "skateboard");
            ```

What is KISS?

"Keep it simple, stupid "

"Keep it simple, silly"

"Keep it short and simple"

"Keep it simple and straightforward"

"Keep it small and simple"

Kelly Johnson

  • Make each program do one thing well
  • To do a new job, build afresh rather than complicate old programs by adding a new 'features'
  • Don't invent complex systems to do simple tasks
"Everything must be made as simple as possible, but not one bit simpler" Albert Einstein

Advantages of the KISS usage

  • The amount of code written reduces
  • The code is easier to read and understand
  • The code is not overloaded with redundant logic
  • It's easier to support and to develop further

Examples of the usage:

  • Application design
  • Application development

In design:

  • it makes no sense to implement additional functions that will not be used at all or their use is extremely unlikely
  • do not overload the interface with options that most users will not need
  • it makes no sense to implement complex business logic that takes into account absolutely all possible behaviors of the system, user and environment

In development:

  • it makes no sense to increase the level of abstraction indefinitely
  • it makes no sense to include redundant functions in the project that may someday be needed (here KISS intersects with the YAGNI principle)
  • you should not include a huge library if you only need a couple of functions from it
  • decomposing something complex into simple components is an architecturally correct approach (here KISS has something in common with DRY)
  • absolute mathematical accuracy or extreme precision are not always needed, and the data can and should be processed with the accuracy that is sufficient for a qualitative solution of the problem
